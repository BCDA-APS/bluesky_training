Using git for version control#

While version control is optional, it is _highly recommended [1] [2] that you place your bluesky instrument directory under some form of software version control. [3] At minimum, this can provide some form of backup protection. It also helps others to collaborate with similar bluesky instruments by sharing your instrument’s implementations.

There are many guides available online to learn about git and GitHub or GitLab. This document will not attempt to cover that material. Rather, this document will describe the steps to place the bluesky directory and its contents under git version control and push to a GitHub repository.

[1]

Why use version control? https://www.git-tower.com/learn/git/ebook/en/desktop-gui/basics/why-use-version-control

[2]

Why use version control? https://about.gitlab.com/topics/version-control/#why-use-version-control

[3]

software version control guide https://www.software.com/devops-guides/version-control-guide

Overview#

  1. Identify the web host for the repository

    1. Possible web hosts include:

      1. GitHub

      2. APS GitLab

      3. others not described here

    2. Use or create an organization for the beamline.

    3. Create a new repository for the bluesky directory

  2. Make the bluesky/ directory into a git repository - either

    • clone the repository from the web host

    • Use `git init <git-guides/git-init>`__ in the bluesky/ directory

  3. Create a `.gitignore file <https://git-scm.com/docs/gitignore>`__, such as this example

  4. Add content and commit with a meaningful commit message (see these hints for relevant prefixes)

  5. Push content to web host

GitHub#

Most beamlines will use GitHub to host their bluesky repositories. These are publicly-accessible (read-only) so do not place any content there that should not be publicly-accessible.

See this document to learn more about:

  • GitHub organizations

  • GitHub repositories

GitHub organizes repositories using this pattern: https://github.com/ORGANIZATION/REPOSITORY

In ORGANIZATION is either a single user’s account or a named orgnization (created by some GitHub user).

Note that GitHub now uses multi-factor authentication (MFA).

Beamline GitHub organizations#

Use GitHub organizations to provide version control for various software projects used by a beamline. This naming convention is recommended for the beamline organization:

  • create a GitHub organization with a name like this : APS-SSS-GGG

  • SSS : sector, beamline, and station (such as 32IDC)

  • GGG : operating group (such as MIC for the XSD/Microscopy group)

  • Within each organization, create a repository : bluesky-INSTRUMENT

  • INSTRUMENT : the name of the sintrument, such as USAXS

APS GitLab#

When there is content that requires access restrictions (thus, not publicly-accessible), the APS GitLab server, managed by the APS IT group, is available to host that content. The server provides online help comparable to that provided by GitHub.

GitLab organizes repositories using this pattern: https://git.aps.anl.gov/ORGANIZATION/SUBGROUP/REPOSITORY

The SUBGROUP level is optional.
